3. Architecture of each solution

A. Plain containers dev-only

OCI container with namespaces, cgroups, seccomp, dropped caps. All tenants share the one host kernel; isolation rests entirely on kernel correctness.

┌──────────┐ ┌──────────┐  per-tenant
│ MCP A    │ │ MCP B    │  containers
├──────────┴─┴──────────┤
│   SHARED HOST KERNEL   │ ◄ one escape = all tenants
└────────────────────────┘

Use: dev wiring only. Reject for untrusted code (HC-3).

B. gVisor (runsc) dev backend

Workload runs atop the Sentry, a user-space kernel that services syscalls, so it rarely touches the host kernel. Much smaller surface than plain containers, far lighter than a VM.

┌──────────┐
│  MCP svr │
├──────────┤
│  Sentry  │ ◄ user-space kernel
├──────────┤
│host kernel│ (filtered surface)
└──────────┘

Use: default local/dev backend + defense-in-depth in prod.

C. microVM — Firecracker / Kata production

Each server runs in its own lightweight VM with a dedicated guest kernel, started by a minimal VMM on KVM (~125 ms boot). Orchestrated via Kata RuntimeClass on k8s; warm pools + scale-to-zero manage cold-start/cost.

┌──────────┐ ┌──────────┐
│ MCP svr  │ │ MCP svr  │
│GUEST KRNL│ │GUEST KRNL│ ◄ own kernel
├──────────┤ ├──────────┤
│Firecrackr│ │Firecrackr│  (KVM)
├──────────┴─┴──────────┤
│       HOST KERNEL      │
└────────────────────────┘

Use: production. Strongest practical isolation; cost/perf premium is the deliberate HC-3-over-SG-4 trade.

D. Serverless containers build-vs-buy

A managed platform runs each workload in its own isolation unit (Fargate & Fly = Firecracker microVMs; Cloud Run gen2 = microVM). VM-grade isolation + auto-scaling without operating the substrate.

┌──────────┐
│  MCP svr │
├──────────┤
│ platform │ ◄ provider microVM
│ isolation│
└──────────┘ (you don't run host)

Use: least ops, less control/portability; cost & cold-start vary by vendor.

Recommendation

Make the sandbox runtime pluggable (a SandboxRuntime interface) with swappable backends:

  • Dev / local / CI → gVisor (Docker Desktop, or minikube + gVisor addon). No KVM, fast loop, faithful to our defense-in-depth layer.
  • Production → microVM (Firecracker/Kata) on KVM-capable nodes. The hard-constraint (HC-3) isolation boundary.
  • Fidelity check → real microVM path in a KVM-enabled Linux VM on your M4 (nested virt) or a remote host; pass the adversarial suite (US4 / SC-002–003) there before any production claim.

This satisfies HC-3 in production while keeping development fast and cheap — consistent with research R1.
